# PR review self-checklist

Answer each YES/NO. Any NO means the verdict is not ready to return.

1. [ ] I read the work order / task statement BEFORE the diff, and listed its
   pinned IDs and allowed files.
2. [ ] Every file in the diff maps to the declared scope, or is listed as an
   out-of-scope finding (schemas, guardrails, settings, CI flagged
   specially).
3. [ ] For every schema edit: mutation regression tests exist in the same
   revision that assert the new form validates AND the previously-invalid
   form still rejects.
4. [ ] For every schema edit: instances, runner, and CHANGELOG move in the
   same revision, and the changelog entry cites the motivating friction or
   finding ID.
5. [ ] Every computed artifact touched (scorecard, gate report, coverage
   numbers) was regenerated by its script — no hand-edited numbers — and I
   know the reproduce command.
6. [ ] No grader/marker/threshold change lands in the same PR as transcripts
   or data that newly pass under it.
7. [ ] I re-ran the relevant tests/scripts myself and quoted the command +
   result; I did not rely on the author's summary or "CI is green" alone.
8. [ ] I re-checked at least one headline claim (byte-identical, count-is-0,
   no-references-remain) with a fresh grep/command.
9. [ ] No FAIL rows, regression floors, or open-friction entries were
   deleted or softened without a stated reason.
10. [ ] Commit plan uses explicit paths with a staged-file-count assertion
    (no blanket `add -A` into mixed commits).
11. [ ] The verdict follows RUBRIC-pr-review's rule mechanically; findings
    are ranked most-severe first with file:line and a concrete failure
    scenario each.
12. [ ] My review states what I could NOT verify (environment limits, missing
    fixtures) instead of scoring those points on trust.
