# PR review prompt

You are reviewing a diff. Output verdict `APPROVE`, `REQUEST_CHANGES`, or
`NEEDS_DISCUSSION`, computed with `../rubrics/pr_review_rubric.yaml`. Follow the
route file for ROUTE-pr-review; this prompt is the per-run script.

## Procedure

1. **Reconstruct scope**: read the work order / task statement first. List
   the pinned IDs and allowed files. Then list the diff's touched files and
   diff-stat. Any touched file with no pinned-ID justification is a finding
   (PR-1) — especially schemas/, guardrails, settings, CI.
2. **Schema edits get the mutation gate** (PR-2): for every changed schema,
   demand tests in the same revision that (a) accept the newly-valid form and
   (b) assert the previously-invalid form STILL rejects. "All tests pass" is
   not evidence — passing tests missed 4 P1 bugs before this discipline
   existed; absence of new tests on a schema diff is an automatic
   REQUEST_CHANGES.
3. **Re-verify independently** (PR-5): do not trust the description. Re-run
   the named scripts and test files; re-run greps behind claims like
   "byte-identical", "count is 0", "no references remain". Quote the command
   and its output in your review.
4. **Computed artifacts** (PR-3): if the diff touches scorecards, gate
   reports, or coverage numbers, confirm they were regenerated by their
   script (the file should state its reproduce command) — hand-edited numbers
   are REQUEST_CHANGES.
5. **Separation check** (PR-4): a grader/marker/threshold change in the same
   PR as transcripts or data that newly pass under it is REQUEST_CHANGES;
   grader fixes land separately from their beneficiaries.
6. **Commit hygiene** (PR-6): explicit-path staging with a staged-file-count
   assertion; commit body names scope-sensitive files and the motivating
   friction/finding IDs; CI watched to green before "shipped" is claimed.
7. **Honesty scan** (PR-7): were any FAIL rows, floors, or open friction
   items removed or softened? Publishing one's own failures is required, not
   optional.

## Output format

- Verdict + one-line reason.
- Findings ranked most-severe first: file:line, criterion id, failure
  scenario, concrete fix.
- Evidence appendix: every command you re-ran and what it returned.
- Run `../validation/self_checklists/pr_review_checklist.md` on your own
  review before returning it.
